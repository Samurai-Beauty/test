- 版数: v1.0(2026-07-22)

- 対象者: アルバイト含む全従業員
- 実装・テスト担当: Claude Code

## 1. 目的とスコープ

### 1.1 目的
Squareの勤怠タイムカードを起点に、勤怠集計 → 給与計算 → 明細発行 → 振込データ作成までを自動化し、手動計算を廃止する。

### 1.2 基本方針(最重要)
- **AIシステムの役割は「集計・検算・データ生成」に限定する**
- **税・社会保険・雇用保険の確定計算はMFクラウド給与のエンジンに委ねる**(料率改定リスクを自前で負わない)
- **振込の実行は必ず人間が行う**(システムは全銀データ生成まで)
- 各工程に承認ゲートを設け、承認なしに次工程へ進まない

### 1.3 スコープ外
- 税額・社保料率の自前計算(MF給与に委譲)
- 振込の自動実行
- 賞与・年末調整(将来フェーズ)

## 2. 全体アーキテクチャ

```
[Square Labor API]
   │ ①タイムカード取得(自動)
   ▼
[集計エンジン(本システム)]
   │ ②残業・深夜・休日の分類、丸め、休憩控除
   ▼
[集計表] ──→ ★承認ゲート①(サムライ社長が確認)
   │
   ▼
[MF給与インポートCSV生成]
   │ ③手動インポート(Phase1)/ Claude in Chrome(Phase2)
   ▼
[MFクラウド給与] … 税・社保・雇用保険を自動計算
   │ ④計算結果CSVをエクスポート
   ▼
[検算エンジン(本システム)] ──→ ★承認ゲート②(差異ゼロ確認)
   │
   ▼
[給与確定] → Web明細自動配信(MF給与機能)
   │
   ▼
[全銀FBデータ出力] ──→ ★承認ゲート③ → 人間が振込実行
```

### 2.1 コンポーネント一覧

| コンポーネント | 実装 | 役割 |
|---|---|---|
| fetch | Python(Square API) | タイムカード・時給マスタ・チームメンバー取得 |
| aggregate | Python | 日次分割・労働時間分類(所定内/時間外/深夜/休日) |
| rules | YAML設定 | 給与規程のルール定義(締め日・丸め・割増率) |
| report | Python | 承認用の集計表出力(Markdown + CSV) |
| export_mf | Python | MF給与インポート形式CSVの生成 |
| verify | Python | MF給与の計算結果と独立検算の突合 |

## 3. 前提・環境情報

### 3.1 Square
- 接続方法: Square API(Access Token、環境変数 `SQUARE_ACCESS_TOKEN`)
- 必要スコープ: `TIMECARDS_READ`, `EMPLOYEES_READ`(読み取り専用トークンを発行すること)
- Location ID:
  - 西新宿本店: `LAC6K4H5FG5RV`
  - 新宿三丁目: `L62PANFASQNVH`
  - 渋谷東: `L95B23P42GDEJ`
- 使用API:
  - `POST /v2/labor/timecards/search`(searchTimecards): 打刻・休憩含むタイムカード取得
  - `GET /v2/labor/team-member-wages`(listTeamMemberWages): 時給マスタ
  - `GET /v2/labor/workweek-configs`(listWorkweekConfigs): 週起算日
  - `POST /v2/team-members/search`: 氏名・在籍情報

### 3.2 MFクラウド給与
- MCP / 公式APIなし → CSVインポート方式
- インポートCSVの正確な列仕様は **画面からテンプレートを取得して確定する(TBD-1)**
- 列マッピングは `config/mf_import_mapping.yaml` に外部化し、コード変更なしで調整可能とする

### 3.3 タイムゾーン
- Square APIはRFC3339(UTCまたはオフセット付き)で返す
- **内部処理はすべて `Asia/Tokyo` に変換してから行う**(深夜判定・日次分割に直結するため厳守)

## 4. 給与計算ルール仕様

### 4.1 設定ファイル `config/payroll_rules.yaml`

```yaml
period:
  closing_day: 99        # TBD-2: 締め日(99=月末)
  payment_day: 99        # TBD-2: 支払日
timezone: Asia/Tokyo
rounding:
  unit_minutes: 1        # TBD-3: 丸め単位(1分推奨。労働者不利の切り捨ては違法リスク)
  method: none           # none | up(労働者有利方向のみ許可)
workweek:
  start_day: MON         # SquareのworkweekConfigsと整合させる(TBD-4)
legal_holiday:
  mode: weekly_one       # 週1日の法定休日
  day: SUN               # TBD-5: 法定休日の曜日 or シフト起点で判定
premium_rates:
  overtime: 1.25         # 時間外(日8h超 or 週40h超)
  overtime_over_60h: 1.50 # 月60時間超の時間外(中小企業も2023年4月から適用)
  late_night: 0.25       # 深夜割増(22:00–05:00)※加算分
  legal_holiday: 1.35    # 法定休日
night_hours:
  start: "22:00"
  end: "05:00"
```

### 4.2 分類アルゴリズム(aggregate)

1. **日次勤務の確定**: タイムカードを1勤務単位とし、暦日を跨ぐ継続勤務は**始業日の勤務**として扱う(労基法の解釈に準拠)
2. **休憩控除**: `breaks[]` の合計を実労働時間から控除。休憩が勤務時間の範囲外にある場合は警告を出しエラーリストへ
3. **日次時間外**: 実労働時間のうち8時間超過分を時間外へ
4. **週次時間外**: 週起算日基準で週40時間超過分を時間外へ。**日次で既に時間外計上済みの時間は除外し二重計上を防止する**
5. **深夜時間**: 勤務時間と22:00–05:00(翌日跨ぎ含む)の重なりを分単位で算出。所定内/時間外いずれとも重複しうる(深夜は加算割増)
6. **法定休日**: 法定休日の労働は全時間を休日労働へ(時間外とは排他)。休日かつ深夜は 1.35 + 0.25 = 1.60
7. **月60時間超**: 月間の時間外累計が60hを超えた分は割増率1.50を適用
8. **丸め**: 分類後の各カテゴリ合計に対して設定の丸めを適用

### 4.3 賃金計算(検算用の独立計算)

- 時給者: `時給 × 各カテゴリ時間 × 割増率` の合計(時給はSquare `listTeamMemberWages` を第一ソース、MF給与マスタと不一致なら警告)
- 月給者: 基準内賃金 ÷ 月平均所定労働時間 で時間単価を算出(TBD-6: 月給者の有無・固定残業代の有無を確認)
- ここでの計算は**検算目的**。支給確定額はMF給与の出力を正とする

## 5. 承認ゲート仕様

| ゲート | タイミング | 承認者 | 確認内容 |
|---|---|---|---|
| ① | 集計表生成後 | サムライ社長 | 人別の総労働・残業・深夜・休日時間、エラーリスト(打刻漏れ等) |
| ② | MF給与計算後 | サムライ社長 | verify結果(差異ゼロ確認)。差異があれば原因特定まで確定禁止 |
| ③ | 全銀データ生成後 | サムライ社長 | 振込総額・件数・口座。承認後に人間が振込実行 |

- システムは承認を記録しない段階(Phase1)では、承認ゲートを「次コマンドを人間が実行する」ことで担保する
- いかなる場合もシステムが自動で振込・給与確定を行ってはならない

## 6. エラーハンドリング仕様

| 事象 | 挙動 |
|---|---|
| OPENのままのタイムカード(退勤打刻漏れ) | 集計から除外し、エラーリストに人名・日付を出力。承認①で必ず提示 |
| 休憩打刻の欠落・範囲外 | 警告リストへ。休憩ゼロのまま集計(過大計上側に倒す=会社不利・労働者有利) |
| 同一人物のタイムカード重複・時間重なり | エラーとして集計停止対象に含め、該当者のみ除外して続行 |
| 時給マスタ欠落 | 該当者を集計表に「単価未設定」と明示 |
| Square APIエラー | 指数バックオフで3回リトライ、失敗時は明示的に異常終了(部分データで出力しない) |

## 7. セキュリティ・個人情報

- `SQUARE_ACCESS_TOKEN` は `.env` で管理し、`.gitignore` に含める(コミット禁止)
- トークンは**読み取り専用スコープ**で発行する
- 出力ファイル(集計表・CSV)には給与情報が含まれるため、`output/` ディレクトリごと `.gitignore` に含める
- テストコードに実在の従業員名・実データを含めない(fixtureはダミーデータのみ)

## 8. 段階導入計画

- **Phase 1(本設計書の実装範囲)**: fetch → aggregate → report → export_mf → verify のCLI一式。MF給与へのインポートは手動。手動計算と1〜2ヶ月並走し差異ゼロを確認してから切替
- **Phase 2**: Claude in ChromeによるMF給与画面操作の型化
- **Phase 3**: 検算レポート定型化、月次KPIワークフロー(エルメ集計)への組込み

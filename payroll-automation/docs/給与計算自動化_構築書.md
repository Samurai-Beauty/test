## 0. Claude Codeへの指示(必読)

1. 本書のタスクを **T1から順番に** 実装し、各タスク完了時にテストを実行してグリーンを確認してから次へ進むこと
2. 設計書のTBD項目は **configのデフォルト値で実装**し、値をハードコードしないこと
3. **実在の従業員データ・本番トークンをテストに使わない**こと。テストはすべてfixture(ダミーデータ)で行う
4. 本番Square APIへの接続テスト(T8)は、読み取り専用トークンが `.env` に設定されている場合のみ実行し、取得件数と期間のみを報告する(個人名・時給をログに出さない)
5. 書き込み系のSquare APIは一切呼ばないこと(使用するのは search/list/get のみ)
6. 迷ったら「会社不利・労働者有利」側に倒す(丸め・休憩欠落時の扱いなど)

## 2. データモデル(models.py)

```python
@dataclass
class Break:
    start_at: datetime  # Asia/Tokyo変換済み
    end_at: datetime | None
    is_paid: bool

@dataclass
class Timecard:
    id: str
    team_member_id: str
    location_id: str
    start_at: datetime
    end_at: datetime | None   # None = OPEN(打刻漏れ)
    breaks: list[Break]
    status: str               # OPEN | CLOSED

@dataclass
class DailyWork:              # 1勤務(始業日基準)の分類結果
    team_member_id: str
    work_date: date           # 始業日
    location_id: str
    total_minutes: int        # 休憩控除後
    regular_minutes: int
    overtime_minutes: int     # 日8h超+週40h超(重複排除済み)
    night_minutes: int        # 22:00-05:00重なり
    holiday_minutes: int      # 法定休日労働
    warnings: list[str]

@dataclass
class MonthlySummary:         # 人別月次(丸め適用済み、月60h超の内訳含む)
    ...
```

金額は `Decimal` を使用し、floatを禁止する。

## 3. 実装タスク

### T1: プロジェクト雛形 + rules.py
- 構成一式、config 3ファイル、`.env.example`、`.gitignore` を作成
- `rules.py`: YAML読込、必須キー検証、`unit_minutes` が1未満・`method` が不正値なら例外

### T2: models.py + タイムゾーン変換ユーティリティ
- RFC3339 → `Asia/Tokyo` 変換関数。UTC入力・オフセット付き入力の両方に対応

### T3: aggregate.py(本システムの核。最重量タスク)
設計書4.2のアルゴリズムを実装。関数を小さく分割すること:
- `deduct_breaks(timecard) -> int`(休憩控除、範囲外休憩は警告)
- `split_daily_overtime(daily_minutes) -> (regular, overtime)`
- `calc_weekly_overtime(week_records) -> 追加overtime`(日次計上分の重複排除)
- `calc_night_overlap(start, end) -> minutes`(日跨ぎ対応)
- `classify_holiday(work_date, rules) -> bool`
- `apply_rounding(minutes, rules) -> minutes`
- `apply_over_60h(monthly_overtime) -> (normal_ot, over60_ot)`

### T4: report.py
- 人別×店舗別の集計表を `summary.md`(承認①提示用)と `summary.csv` に出力
- `errors.md`: OPENタイムカード・休憩警告・単価未設定を人名/日付付きで列挙。**エラーが1件でもあれば集計表冒頭に目立つ形で件数を表示**

### T5: export_mf.py
- `mf_import_mapping.yaml` の列定義に従いCSV生成(仮仕様: 従業員コード, 氏名, 所定内時間, 時間外時間, 60h超時間外, 深夜時間, 休日時間)
- 時間の出力形式は mapping 側で `HH:MM` / 十進時間 を切替可能にする(MF給与テンプレ確定後に合わせるため)

### T6: verify.py
- 入力: MF給与からエクスポートした計算結果CSV(fixture仕様で仮実装)
- 独立計算(時給×時間×割増率)と突合し、人別差異を `verify_report.md` に出力
- 差異ゼロなら `RESULT: OK`、1円でも差異があれば `RESULT: NG` と原因候補を出力

### T7: fetch.py + square_client.py
- `searchTimecards`(期間・location絞り込み、cursorページネーション)、`listTeamMemberWages`、`listWorkweekConfigs`、`searchTeamMembers`
- リトライ: 429/5xxで指数バックオフ3回。失敗時は部分出力せず異常終了
- 取得結果は `output/{period}/raw_timecards.json` に保存(再集計時にAPI再呼出し不要にする)

### T8: cli.py 結合
```
payroll fetch     --period 2026-07
payroll aggregate --period 2026-07   # raw JSONから集計、summary/errors出力
payroll export-mf --period 2026-07   # 承認①後に人間が実行
payroll verify    --period 2026-07 --mf-csv <path>
```
コマンド分離自体が承認ゲートの実装である(自動連続実行を作らないこと)。

## 4. テスト仕様(pytest)

### 4.1 aggregate(最重要・全ケース必須)

| # | ケース | 期待値 |
|---|---|---|
| A1 | 9:00-18:00 休憩60分 | 所定内480分、残業0 |
| A2 | 9:00-20:00 休憩60分 | 所定内480、残業120 |
| A3 | 15:00-23:30 休憩30分 | 深夜90分(22:00-23:30)を含む |
| A4 | 21:00-翌6:00 休憩60分(夜勤) | 始業日の勤務として計上、深夜=22:00-翌5:00の実労働重なり |
| A5 | 休憩3回(合計90分) | 全休憩が控除される |
| A6 | 週6日×7h勤務(日次残業なし) | 週40h超の2hが残業になる |
| A7 | 週5日×9h勤務 | 日次残業5hのみ。週40h判定で**二重計上しない**(週次追加残業0) |
| A8 | 法定休日に8h勤務 | holiday=480、overtime=0(排他) |
| A9 | 法定休日 20:00-24:00 | holiday=240、うち深夜120(割増1.60対象の内訳が出る) |
| A10 | 月間残業65h | normal_ot=60h、over60_ot=5h |
| A11 | 丸め: unit=15/method=up、実働7h50m | 8h00m(労働者有利方向) |
| A12 | 丸め: method=none | 1分単位そのまま |
| A13 | end_at=None(OPEN) | 集計除外+エラーリスト掲載 |
| A14 | 休憩がstart_at前に開始 | 警告+休憩控除なしで集計 |
| A15 | 同一人物の時間帯重複タイムカード | 該当者除外+エラー、他の従業員は正常集計 |
| A16 | UTC入力(例: 2026-07-01T13:00:00Z=JST22:00) | 深夜判定がJST基準で正しい |
| A17 | 月末締め: 6/30 22:00-7/1 6:00 の勤務 | 始業日基準で6月分に計上 |

### 4.2 verify

| # | ケース | 期待値 |
|---|---|---|
| V1 | 完全一致 | RESULT: OK |
| V2 | 1名だけ100円差異 | RESULT: NG、該当者と差額を特定 |
| V3 | MF給与側に存在しない従業員 | NG、欠落として報告 |
| V4 | 時給1,200円×残業2h | 独立計算=1200×2×1.25=3,000円(Decimal検証) |

### 4.3 fetch(httpxモック)
- F1: cursorページネーション2ページを結合できる
- F2: 429→リトライ→成功
- F3: 3回失敗→例外で異常終了、outputに部分ファイルが残らない

### 4.4 export_mf
- E1: mappingのHH:MM形式指定で `08:30` と出力される
- E2: 十進形式指定で `8.5` と出力される
- E3: mapping変更のみで列順が変わる(コード変更不要の確認)

## 5. 受け入れ基準

1. `pytest` 全件グリーン(カバレッジ: aggregate.py 95%以上)
2. fixtureの1ヶ月分ダミーデータで `fetch(モック)→aggregate→report→export-mf→verify` が一気通貫で動作
3. エラー系(A13-A15)が承認①の資料(errors.md)に必ず現れる
4. `git grep` で時給・料率・締め日のハードコードが存在しない(すべてconfig参照)
5. README.mdに月次運用手順(コマンド順+承認ゲートの位置)が記載されている

## 6. 実装完了後の運用テスト(人間と実施)

1. 読み取り専用のSquare本番トークンを `.env` に設定
2. `payroll fetch --period 2026-07` で実データ取得(直近締め期間)
3. 集計表を手動計算と突合 → 差異があればルール設定(TBD-2〜5)を修正
4. 1〜2ヶ月並走で差異ゼロを確認してから正式切替

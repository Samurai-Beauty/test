概要

システム概要・機能一覧

このシステムはHTMLファイル1つで動作する美容室向けシフト管理PWA（スマホアプリ化可能なWebアプリ）です。
サーバーは不要で、すべて無料サービスを組み合わせて構築します。

機能内容利用者
シフト希望提出スタイリストが毎月のシフト希望を入力・送信スタイリスト
シフト管理・確定希望をもとに管理者がシフトを編集・確定本社・店舗管理者
シフト閲覧確定シフトをカレンダー表示（店舗別）全員
お知らせ本社・管理者からのお知らせ投稿・閲覧全員
スタイリスト管理スタッフの追加・削除・店舗割り当て本社・管理者
資料共有PDFなどのファイルをアップロード・共有全員
売上管理CSVアップロードによる売上データ集計本社のみ
ご意見箱スタイリストから本社への匿名投稿全員
🏗️
技術構成：フロントエンド = HTML / CSS / JavaScript（フレームワーク不使用）、データ保存 = Supabase（PostgreSQL + ファイルStorage）、ホスティング = GitHub Pages

STEP 2

GitHubリポジトリ作成

約30分
  • 1
    GitHubにログインし、右上「＋」→「New repository」をクリック
  • 2
    Repository name を入力（例：salon-shift）、Publicを選択（GitHub Pages無料公開に必要）
  • 3
    「Add a README file」にチェックを入れて「Create repository」をクリック
  • 4
    ローカルPCにクローン：
    git clone https://github.com/あなたのID/salon-shift.git
    cd salon-shift
  • 5
    元のシステムの index.html をフォルダにコピー
STEP 3

Supabaseセットアップ

1.5〜2時間

最も時間がかかるステップです。丁寧に進めてください。

① プロジェクト作成

  • 1
    supabase.com にログインし「New Project」をクリック
  • 2
    プロジェクト名・パスワード・リージョン（Northeast Asia推奨）を設定して作成（2〜3分かかる）
  • 3
    「Project Settings」→「API」から Project URL と anon public key をメモしておく

② テーブル作成

左メニュー「SQL Editor」を開き、以下を実行：

-- データ保存テーブル作成 CREATE TABLE public.sales_data ( id bigserial PRIMARY KEY, store_key text NOT NULL, filename text, data_json jsonb, uploaded_at timestamptz DEFAULT now(), uploaded_by text ); -- シフト重複防止のUNIQUE制約（必須） ALTER TABLE public.sales_data ADD CONSTRAINT sales_data_store_key_unique UNIQUE (store_key);

③ RLS（行レベルセキュリティ）を無効化

  • 1
    左メニュー「Table Editor」→「sales_data」テーブルを選択
  • 2
    右上の「RLS enabled」ボタンをクリック→「Disable RLS」を選択
⚠️
RLSを無効化しないとデータの保存・読み込みが全て失敗します。必ず実行してください。

④ Storage（ファイル保存）セットアップ

  • 1
    左メニュー「Storage」→「New bucket」をクリック
  • 2
    Bucket name：documents、Public bucket：ONにして作成
  • 3
    SQL Editorで以下を実行してアップロード権限を付与：
    CREATE POLICY "allow all" ON storage.objects FOR ALL TO public USING (bucket_id = 'documents') WITH CHECK (bucket_id = 'documents');
STEP 4

コードカスタマイズ

3〜5時間

index.html をエディタで開き、以下の箇所を上から順に書き換えます。
「Ctrl+F（Mac: Cmd+F）」で検索しながら進めると効率的です。

① Supabase接続情報（必須）

検索ワード：SUPABASE_URL

const SUPABASE_URL = '★ここにProject URLを貼る★'; const SUPABASE_KEY = '★ここにanon public keyを貼る★';

② 店舗情報の設定（必須）

検索ワード：STORE_LABELS

// 店舗のキー名と表示名 const STORE_LABELS = { honsha: '本社', store1: '○○店', // ← 実際の店舗名に変更 store2: '△△店', store3: '□□店', };
💡
店舗のキー名（store1など）はアルファベットで設定します。表示名（○○店）は日本語でOKです。STORE_COLORSで各店舗のカラーも変更できます。

③ スタッフ・パスワードの設定（必須）

検索ワード：STORE_MEMBERS

const STORE_MEMBERS = { honsha: ['管理者A', '管理者B'], store1: ['スタッフA', 'スタッフB', 'スタッフC'], store2: ['スタッフD', 'スタッフE'], store3: ['スタッフF'], };

検索ワード：DEFAULT_PASSWORDS

const DEFAULT_PASSWORDS = { '管理者A': 'Salon.2025', // 任意のパスワードに変更 'スタッフA': 'Salon.2025', // 全員分設定する };

④ FCオーナー（店舗限定管理者）の設定（任意）

検索ワード：STORE_MANAGERS

const STORE_MANAGERS = { store3: ['スタッフF'], // store3のみ管理できるオーナー };

⑤ 初期シフトデータの削除（必須）

検索ワード：seedMay2026IfNeeded

// この関数の中身（サンプルデータ）を空にする function seedMay2026IfNeeded() { // 何も書かなくてOK（空のままにする） }
⚠️
元のシステムには別サロンのサンプルシフトデータが入っています。そのまま使うと他サロンのデータが混入します。必ず空にしてください。
STEP 5

GitHub Pagesで公開

約30分
  • 1
    変更したファイルをGitHubにアップロード：
    git add index.html
    git commit -m "初期設定"
    git push origin main
  • 2
    GitHubのリポジトリページで「Settings」→「Pages」を開く
  • 3
    「Source」で「Deploy from a branch」、Branch で「main」/「root」を選択して「Save」
  • 4
    数分後、https://あなたのID.github.io/salon-shift/ でアクセス可能になる
✅
このURLがシステムのアドレスになります。スマホのホーム画面に追加すればアプリのように使えます。
STEP 6

動作確認チェックリスト

約1時間

本番公開前に以下の項目を順番に確認してください：

  • 本社アカウントでログインできる
  • スタイリストアカウントでログインできる
  • シフト希望を提出できる（スタイリストで）
  • シフトが管理画面に反映されている（本社で）
  • シフトを確定してスタイリスト閲覧画面に表示される
  • お知らせを投稿・閲覧できる
  • スタイリストを追加・削除できる（本社で）
  • 資料をアップロードしてダウンロードできる
  • ページをリロードしてもデータが消えない（Supabase保存確認）
  • スマホからアクセスして正常に表示される
  • FCオーナーがある場合、担当店舗のみ表示される
⚠️
リロード後にデータが消える場合は Supabase のRLS設定かUNIQUE制約が正しく設定されていない可能性があります。STEP 3 を見直してください。
STEP 7

スタッフへの展開

約30分
  • 1
    URLとログイン情報（名前・パスワード）を各スタッフに伝える
  • 2
    スマホのホーム画面への追加を案内：
    iOS（Safari）：共有ボタン →「ホーム画面に追加」
    Android（Chrome）：メニュー →「ホーム画面に追加」
  • 3
    初回ログイン後にパスワードをメモ・保存するよう案内（ブラウザのパスワード保存機能を使うと便利）
✅
パスワードはコード内の DEFAULT_PASSWORDS に直接書くため、スタッフ全員に同じ初期パスワードを設定して、後から変更する運用も可能です。
FAQ

よくあるトラブルと対処法

症状 原因と対処法
保存してもリロードで消える Supabase の RLS が有効のまま。「STEP 3 ③」を確認してRLSを無効化する
「400エラー」が表示される sales_data テーブルに UNIQUE 制約がない。「STEP 3 ②」のSQLを再実行
ファイルアップロードが失敗する Storageのポリシーが設定されていない。「STEP 3 ④」のSQLを再実行
ページが真っ白で表示されない SUPABASE_URL か SUPABASE_KEY が間違っている。コピペミスを確認
ログインできないスタッフがいる STORE_MEMBERS か DEFAULT_PASSWORDS のスペル・全角半角を確認
スマホで画面が崩れる 古いブラウザの場合あり。Safari / Chrome の最新版にアップデート
GitHubにpushできない Gitの認証エラー。Personal Access Token を生成して使用（GitHub Settings → Developer settings）
